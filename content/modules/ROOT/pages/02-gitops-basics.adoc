お客様専用のArgo CDインスタンスがOpenShiftプロジェクト `{user}-argocd` にデプロイされました。これは、OpenShiftが採用しているオペレーターベースのメカニズムの利点の1つです。ユーザーはソフトウェアやソリューションを簡単にデプロイできます。

== Argo CD デプロイメントの確認

OpenShift Web コンソールで、左側のナビゲーションにある _Workloads_ メニューをクリックして開き、次に _トポロジー_ リンクをクリックします。上部で `{user}-argocd` プロジェクトが選択されていることを確認してください。トポロジビューに Argo CD デプロイメントが表示されます。

image::argocd-topology.png[]

[#connect-argocd]
== Argo CDへの接続

Argo CDは初回デプロイ時にデフォルトの `admin` ユーザーとランダムなパスワードを生成しますが、OpenShift認証を使用するように設定することもできます。

このユーザーアカウントを使用して、CLIまたはWebコンソール経由でArgo CDに接続できます。
このワークショップでは、Argo CD Webコンソールのみを使用します。

Argo CDインスタンスのURLを見つける方法はいくつかあります。プロジェクトの _トポロジー_ ビューでは、サービスに関連付けられているRoutes/Ingressesに小さなポップアップアイコンが表示されます。

image::argocd-topology-link.png[]

左側の _プロジェクト_ リンクをクリックし、次のページで _Route_ をクリックすると、すべてのルートが表示されます。

また、`oc` CLI を使用して Argo CD ルートを取得することもできます。

[.console-input]
[source,bash,subs="attributes+,+macros"]
----
oc get route -n {user}-argocd {user}-argo-server -o jsonpath='{.spec.host}{"\n"}'
----

Argo CDコンソールへのアクセス方法は問いません。OpenShiftへのログインに使用したユーザー名とパスワードと同じものを使用してください。

image::argo-cd-login.png[ArgoCDLogin, 600]

認証情報を入力すると、アクセス権限を承認するよう求めるメッセージが表示されます。 "Allow selected permissions" ボタンをクリックしてください。

image::argo-cd-allow-access.png[ArgoCDAuthorize, 600]

ログインすると、次のページが表示されます。これがArgo CDのWebユーザーインターフェースです。

image::argocd-login2.png[ArgoCDLogin2, 600]

=== アプリケーションのデプロイ

Argo CD では、管理対象のマニフェストの集合は `Application` と呼ばれます。

そのため、Argo CD がこれらのマニフェストをクラスターに適用するには、以下のリンクを使用してアプリケーションを定義する必要があります。
link:https://argo-cd.readthedocs.io/en/stable/operator-manual/declarative-setup/#applications[アプリケーション CR (カスタムリソース)^]
クラスターでこれらのマニフェストを適用するには、以下の手順が必要です。


このアプリケーションのデプロイに使用された `Application` マニフェストを確認してみましょう。/content/modules/ROOT/examples/bgd-app.yamlです。そして、これを少し詳しく見ていきましょう。

image::bdg-applicationyaml.png[bgd-applicationyaml, 600]


.link:https://github.com/OpenShiftDemos/openshift-gitops-workshop/blob/{gitops_revision}/content/modules/ROOT/examples/bgd-app.yaml[bgd-app.yaml,window="_blank"]
[source,yaml,subs="+macros,attributes+"]
----
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: bgd-app
spec:
  destination:
    namespace: {user}-bgd
    server: https://kubernetes.default.svc <1>
  project: default <2>
  source: <3>
    path: documentation/modules/ROOT/examples/bgd
    repoURL: https://github.com/OpenShiftDemos/openshift-gitops-workshop
    targetRevision: main
  syncPolicy: <4>
    automated:
      prune: true
      selfHeal: false
----

<1> destination serverはArgo CD が実行されているクラスターの API エンドポイントです。
この場合、クラスターのローカルで解決可能な URL を使用します。
<2> ここでは、Argo CD の `default` プロジェクト (`.spec.project`) にアプリケーションをインストールします。
[NOTE]
Argo CD の `Project` の概念は OpenShift とは異なります。ここでは、アプリケーションを Argo CD の `default` プロジェクト (`.spec.project`) にインストールします。
OpenShift のデフォルトプロジェクト *ではありません* 。

<3> マニフェストリポジトリ、およびその中のYAMLファイルが格納されているパスです。
<4> `syncPolicy` は `automated` に設定されています。これにより、Git リポジトリから削除されたリソースは自動的に削除されますが、リポジトリに保存されている定義から逸脱したリソースは自動的に修正されません。つまり、 `oc` または `kubectl` を使用して手動で行われた変更は *修復* されません。


Giteaの編集機能を利用してサンプルコードを少し変更することでアプリケーションを作成します。

`bgp-app.yaml` を指定して、右上の鉛筆マークにカーソルを合わせると `ファイルを編集` と表示されるのでクリックすると編集画面に遷移します。

image::git-edit01.png[git-edit01, 600]

OpenShiftのネームスペースに `{user}-bgd` を指定します。repoURLとして `https://gitea.apps.cluster-8krr9.dynamic.redhatworkshops.io/user2/openshift-gitops-workshop` を指定します。その際、GiteaのURLの `user2` の部分は、使用しているGiteaのユーザ名にしてください。

image::git-edit02.png[git-edit02, 600]

`変更をプレビュー` をクリックすると、編集箇所が表示されます。

image::git-edit03.png[git-edit03, 600]

正しく入力されていることを確認し `変更をコミット` をクリックします。

image::git-edit04.png[git-edit04, 600]

正しく変更されていることを確認します。これでアプリケーションをデプロイする準備が整いました。

image::git-edit05.png[git-edit05, 600]


`oc apply` コマンドを実行し、OpenShift環境にデプロイします。その際、GiteaのURLの `user2` の部分は、使用しているGiteaのユーザ名にしてください。

=== アプリケーションの変更

Git上でアプリケーションに変更を加えて、ArgoCDがどのように動作するか確認してみましょう。

Giteaで、`bgd-deployment.yaml` を指定して、右上の鉛筆マークにカーソルを合わせると `ファイルを編集` と表示されるのでクリックすると編集画面に遷移します。

image::change-app1.png[change-app1]

/spec/template/spec/containers以下に環境変数としてCOLORを指定する行があります。オリジナルは `blue` となっているので、この部分を `green` に変更します。

image::change-app2.png[change-app2]

`変更をプレビュー` ボタンをクリックし、変更内容を確認した上で、`変更をコミット` ボタンをクリックします。

image::change-app3.png[change-app3]

Argo CDのユーザーインターフェースを見てみましょう。ArgoCDの Automated Sync（自動同期）で、Gitの変更が反映されるまでの時間はデフォルトで**最大3分（180秒）**です。約3分待つと、新しいReplicaSetが作成され、新しいPodが作成される状態を見ることができます。

image::change-argocd1.png[change-argocd1]

再びアプリケーションを見てみましょう。アプリケーションのURLにアクセスすると、先ほど環境変数を `green` に変更した内容が反映されて、バブルの色が緑になっていることを確認できます。

このように、Gitの情報を真として、クラスタが構成されることがわかりました。

image::change-argocd2.png[change-argocd2]

=== 設定のdriftへの対処

本番環境に変更を加えます。本番環境のデプロイメントマニフェストにパッチを適用して、アプリケーション内のバブルの色を緑から赤に変更します。

[.console-input]
[source,bash,subs="attributes+,+macros"]

----
oc -n {user}-bgd patch deploy/bgd --type='json' -p='[{"op": "replace", "path": "/spec/template/spec/containers/0/env/0/value", "value":"red"}]'
----

デプロイメントで新しいポッドのロールアウトが行われるまでお待ちください。

[.console-input]
[source,bash,subs="attributes+,+macros"]
----
oc rollout status deploy/bgd -n {user}-bgd
----

アプリケーションが実行されているブラウザタブを更新してください。赤色のバブルが表示されるはずです。

image::bgd-red.png[BDG red]

Argo CDのWeb UIを確認すると、Argoがアプリケーションを "Out of Sync"と検出していることがわかります。これはGit上の設定と本番環境の際が発生したことを示しています。

image::sync1.png[Out of Sync]

Argo CD を介してアプリを同期するには、以下の方法があります。

* 最初に左上部にある `SYNC` ボタンをクリックします。
* 次に、右側に表示される `SYNCHRONIZE` ボタンをクリックします。

同期処理が完了すると、Argo CDのユーザーインターフェースはアプリケーションが同期済みであることを示します。

image::sync2.png[Fully Synced]

アプリケーションが実行されているタブのページを再読み込みしてください。バブルの色が元の緑色に戻っているはずです。

image::change-argocd2.png[change-argocd2]

上記の例では、手動で同期を行いましたが、自動で行うことも可能です。Argo CD でdriftを自動的に修正するには、`Application` マニフェストの `selfHeal` プロパティを設定します。link:#_deploy_the_application[above]: #_deploy_the_application の例を参照してください。

[.console-input]
[source,yaml,subs="attributes+,+macros"]
----
# bgd-app.yaml
...
spec:
  syncPolicy:
    automated:
      prune: true
      selfHeal: true # Set this to true
----

あるいは、今回のように、事後に以下のコマンドを実行することによっても可能です。

[.console-input]
[source,bash,subs="attributes+,+macros"]
----
oc patch application/bgd-app -n {user}-argocd --type=merge -p='{"spec":{"syncPolicy":{"automated":{"prune":true,"selfHeal":true}}}}'
----
